What is logging good for

According to the different sources, we can identify several ways people want to access the data of the log file:

  • instant reading (output in a console for example) : tracing, live debugging...
  • delayed reading (consult the log file at the end of execution for example) : performance analysis, ...
  • error diagnostic (dump a file if an error occured, like a crash) : post-mortem analysis, ...

We can also identify different purpose of logging:

Regarding the way data needs to be filtered:

  • filtering is done by the application itself at compile time
  • filtering is done by the application itself at runtime time
  • the application do no filtering, it is optionaly done by another tool when consulting the log data (log viewer, SQL request,...)

Regarding the way data needs to be saved:

  • no need to save data (console output...)
  • save data only on some condition
  • always save data

People also have many different needs regarding the place log data should be saved:

  • several files for a single process (error.log, warning.log, ...)
  • one file for one single process
  • one file for all the instances of a same application (webserver)
  • centralized logging from different sources (database, system logger, logserver)

special considerations regarding postmortem logs

Generating a postmortem log can be verry difficult, because it often has to be done in the context of a crash, that is a dying application. The memory can be completely corrupted, or it can be full and the system is running under very low ressource, etc... Handling such problems should be of the responsibility of the crash handler (like setting up a new stack in case of a stack overflow), but a well designed logging library should allows she crash handler to dump as much information as possible, without running into trouble.

special considerations regarding log saving and sending

When generating post execution logs in the context of an open source software or in teh context of debugging a closed source software within the developpement team, there is almost no restriction regarding the type of information that can be put in the log file.

However, people developping closed source software may worry about the type of information that is dumped in the log file at the client side. In particular, they might not appreciate that anyone can read all the logged messages, as well as the source file and function names, etc...

The perfect logging library should also have this in mind, and provide an easy way to dump the data in a non human readable format.
